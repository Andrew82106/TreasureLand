# 《万物之岛》CI 与自动化测试宪法 v0.1

> 文档性质：项目级质量约束，优先于具体 CI Workflow 与单个测试脚本  
> 适用范围：设计规则、Godot 工程、游戏数据、美术资源、存档、构建与发布流程  
> 当前引擎基线：Godot 4.7 stable  
> 生效原则：先依据本宪法完成“测试影响审查”，再编写或修改 `.github/workflows/` 下的具体配置  
> 核心目标：项目持续扩张时，新增内容不破坏既有规则，修复过的问题不再复发，自动化门禁不会反过来绑架设计

---

## 1. 宪法地位与适用顺序

1. 游戏总文档和各模块设计文档决定“游戏应该是什么”。
2. 本文档决定“哪些承诺必须被自动验证，以及如何验证”。
3. 测试代码负责表达可执行的验收条件。
4. GitHub Actions Workflow 只负责在规定时机、规定环境中执行这些测试。
5. CI 配置不得自行创造玩法规则，也不得用当前代码行为覆盖设计文档中的正确规则。

当四者发生冲突时，必须先判断冲突来源：

- 设计已经正式修改：同一次变更中更新设计文档、实现和测试。
- 实现偏离设计：修改实现，并为该问题补回归测试。
- 测试写错：修正测试，提交说明中必须解释原断言为什么不能代表设计。
- CI 环境问题：修正环境或隔离不稳定测试，不得为了通过而删除有效业务断言。

本文使用以下约束词：

- **必须**：不满足时不得视为可上传或可合并。
- **应当**：原则上必须满足；例外需要在提交说明中写明理由。
- **可以**：按变更风险选择。

## 2. CI 的目标与非目标

### 2.1 CI 必须守住的目标

CI 应持续回答以下问题：

1. 项目能否被指定版本的 Godot 正常导入和解析？
2. 核心规则、资金、物品、时间和状态是否守恒？
3. 修复过的缺陷是否再次出现？
4. 跨模块闭环是否仍能完成？
5. 随机系统是否可复现，统计结果是否落在设计允许区间？
6. 关键界面结构和交互入口是否仍然存在且可用？
7. 正式资源是否能够加载，动画图集与约定是否匹配？
8. 具备导出条件后，目标平台构建是否成功？

### 2.2 CI 不负责替代的工作

以下问题不能仅由 CI 判定：

- 游戏是否好玩、节奏是否有吸引力。
- 美术是否足够精美、角色是否有魅力。
- 动画停顿是否“有智慧”、NPC 台词是否自然。
- 某个经济倍率是否最终合理。
- 地图引导是否符合真实玩家直觉。

这些内容必须通过试玩、视觉评审和数据复盘判断。CI 可以检查资源完整性、统计区间和结构约束，但不能把主观质量伪装成一个布尔断言。

## 3. CI 配置范围

### 3.1 当前必须纳入

- Godot 项目导入、脚本解析和主场景加载。
- 纯规则与数据不变量测试。
- 模块级场景测试。
- 核心跨模块烟雾测试。
- 经济、牌局、赛事等高风险随机系统的固定种子压力测试。
- 关键 UI 的结构、文案约束、按钮可用性和溢出边界测试。
- 素材路径、图集尺寸、动画名称和必要帧是否完整。

### 3.2 条件具备后纳入

- 存档往返、旧版本迁移和坏档保护测试。
- Windows 导出与启动检查。
- Release 安装包生成、校验和与产物上传。
- 性能基线、加载耗时和内存回归。
- 多分辨率 UI 截图对比。

### 3.3 默认不纳入主门禁

- 依赖人工账号、真实用户存档或本机绝对路径的测试。
- 依赖公网实时数据且没有固定快照的测试。
- 未固定字体、引擎版本、平台和分辨率的像素级截图对比。
- 长时间运行但没有明确风险目标的穷举测试。
- 只验证日志中出现某句话、却不验证真实状态的脆弱测试。

## 4. 测试分层

每个测试必须属于以下层级之一。新功能不要求机械地覆盖所有层级，但必须覆盖与风险相匹配的层级。

| 层级 | 名称 | 主要验证 | 是否阻断合并 | 目标时长 |
|---|---|---|---:|---:|
| L0 | 导入与解析 | Godot 版本、资源导入、脚本解析、主场景加载 | 是 | 1 分钟内 |
| L1 | 规则与契约 | 配方、牌型、价格、时间、守恒、输入输出边界 | 是 | 单项数秒 |
| L2 | 模块场景 | 一个模块从入口到结算的完整行为 | 是 | 单项 30 秒内 |
| L3 | 跨模块烟雾 | 发现—图鉴—合成—服务—活动—结算闭环 | 是 | 3 分钟内 |
| L4 | 统计与压力 | 状态机收敛、随机分布、经济期望、长局守恒 | 核心系统是 | 5 分钟内 |
| L5 | UI 结构与交互 | 关键节点、按钮、焦点、可滚动区、信息不重复 | 是 | 2 分钟内 |
| L6 | 视觉回归 | 固定条件下的截图差异与人工审阅 | 初期否 | 按需 |
| L7 | 构建与产物 | Windows 导出、启动、文件完整性 | 发布时是 | 10 分钟内 |

规则测试与视觉测试必须分离。界面好不好看由视觉评审决定；但按钮被遮挡、内容重复、结算区不可见等可客观判定的问题必须进入 L5。

## 5. 哪些修改必须新增或更新测试

### 5.1 必须新增回归测试

每一个经过确认并修复的玩家可见缺陷，必须先提炼出“修复后永远成立的条件”，再加入最小回归测试。例如：

- 玩家或 NPC 重复扣款：测试每席本轮投入与底池守恒。
- 玩家退契后无资格者参与分池：测试席位资格与边池名单。
- NPC 退契后仍继续思考或说话：测试退契状态不再进入行动队列。
- 结算页溢出：测试固定舞台、滚动区域与结果挂载位置。
- 同一角色一次行动说两句话：测试一个行动编号最多产生一次主气泡。

回归测试名称或失败信息必须能指向原问题，不能只写“结果错误”。

### 5.2 下列变更必须评估并更新现有测试

| 变更类型 | 最低测试要求 |
|---|---|
| 新增玩法模块 | L1 + L2；接入全局状态后补 L3 |
| 修改核心规则 | 更新规则文档、L1、相关 L2；影响跨模块时更新 L3 |
| 修改经济数值 | 守恒测试、边界测试、来源/去向测试；概率收益补 L4 |
| 修改随机算法或权重 | 固定种子复现测试 + 统计区间测试；记录样本量 |
| 修改时间推进或日结 | 连续delta、帧率误差、暂停/恢复、固定活动单次提交、跨边界顺序、日终停驻、离线停止和幂等性测试 |
| 修改图鉴、商店或合成 | 开局三根源、四阶规模、层间关系覆盖公式、组合与输出唯一性、完整可达性、原子扣款、重复复查、失败反馈和跨模块一致性测试 |
| 修改牌会行动逻辑 | 顺序、轮次结束、跟契差额、全投、退契、主池/边池和资金守恒测试 |
| 修改 UI 布局或交互 | L5；重要页面同时生成 L6 评审截图 |
| 替换角色或物品素材 | 路径、尺寸、透明度、图集切片、动画名和占位图检测 |
| 修改存档字段 | 默认值、保存—读取往返、旧档迁移、未知字段容错测试 |
| 升级 Godot 版本 | L0 至 L5 全量运行；重新确认截图基线与导出模板 |
| 修改 CI/测试基础设施 | 以最小示例验证成功路径和故意失败路径；检查权限与超时 |

### 5.3 通常不要求新增游戏测试

- 只修改错别字且不改变玩家可见规则含义。
- 只增加内部注释。
- 只调整不会被程序读取的设计排版。

但即使属于以上情况，上传前仍必须执行第 10 节的测试影响审查，并明确选择“无需改测试”的理由。

## 6. 各模块最低测试契约

| 模块 | 必须长期守住的不变量 |
|---|---|
| 万物合成 | 开局只有水火土；四阶规模严格为3→3→11→55，共72个万物与69条关系；层间新增关系覆盖率按向上取整公式为100%、70%、40%；输入对与输出均唯一；完整谱系可达；万物永久不消耗；新组合试验费原子结算；重复复查免费；四阶终点不得扣款或写入记录；成功与失败都留下可解释记录 |
| 命运牌会 | 六席统一顺序；投入只扣差额；行动轮有限收敛；退契者不再行动；主池、边池、返还、服务费严格守恒；摊牌结果完整可见 |
| 金贝与经济 | 金贝不得无来源增加或无去向消失；交易前后能解释；净资产口径唯一；活动限额和风险上限生效 |
| 时间与日程 | `world_active`按连续delta推进且不依赖输入；`ui_paused`、失焦、最小化和离线不推进且恢复时不补算；`activity_snapshot`只在离场提交一次固定耗时；90/120/150秒档只改变自然流速；跨边界事件顺序稳定；第16潮刻进入日终停驻；日结只执行一次且跨日刷新不重复发奖或扣款 |
| 逐风竞速 | 每日四场赛事按日种子确定；阵容、天气、开盘票池与赛事随机种子可存读且不重掷；同一时段同一赛事只能结算一次；当前预计赔率与封盘赔率均受真实概率返还上限约束；NPC追加票与玩家票据进入封盘快照；四段保存八兽完整顺序；造物不改变赛果、票池或赔率；投入上限生效；长期期望不产生无风险指数增长 |
| NPC 与社交 | 6名核心人物各自具备8条普通对白、3条关系反应、1个基础委托与四时段日程；5—8名环境居民不建立长期关系线；关系严格限制在-100—100；最近记忆最多5条、长期记忆最多2条且行为影响可查询；公开事实、来源消息与人物观点标注来源和有效期；重复询问、分享与交付不重复发奖；深入交谈只结算一次固定潮刻；离场者不继续行动或发言；疏远与离店不关闭公共服务；认识状态、话题、委托、记忆与影响数值可存读 |
| 地图与移动 | 出生点可达核心入口；交互范围正确；站立与步行只产生自然走时、不追加固定扣时；地图打开暂停；快速前往恰好提交0.25潮刻并正确处理边界；遮挡和碰撞不封死路线 |
| 商店与交易 | 不直接解锁或出售谱系万物；服务购买原子完成；提示与优惠次数可追溯 |
| 图鉴与凭证 | 全模块读取同一永久发现集合；每项发现保存首次来源、日、潮刻、输入、关系和因果；版本2及更早布尔发现可迁移；万物不会因使用减少；已验证与未尝试关系统计准确；详情预填造化盆不扣款或写历史；永久万物、有限凭证、鱼获箱与海生记录四类数据不混排；塔层只点亮一次；有限凭证次数不得为负；物品 ID 稳定 |
| UI 与美术资源 | 关键内容不被遮挡；信息不重复；正式素材能加载；像素角色图集符合帧规范；缺失素材时明确失败而非静默替换 |
| 存档 | 同版本往返一致；新字段有默认值；旧版本可迁移；损坏数据不会覆盖最后一个有效存档 |

新增模块时，必须在本表中增加该模块的最低测试契约，或在对应设计文档中建立同等明确的测试契约并从此处链接。

## 7. 测试编写细则

### 7.1 测试结构

每个测试应当清楚区分：

1. **准备**：创建干净状态并明确初值。
2. **执行**：只执行要验证的行为。
3. **断言**：检查玩家可见结果和关键不变量。
4. **清理**：删除临时文件、测试存档和生成资源。

一个测试失败时，失败信息必须说明“什么规则被破坏、期望什么、实际是什么”。

### 7.2 独立性

- 测试不得依赖执行顺序。
- 测试不得读取真实玩家存档或复用真实牌局记录。
- 每项测试必须自行创建状态，并在需要时关闭正式记录写入。
- 临时输出必须写入独立测试目录，完成后清理。
- 测试不得依赖开发者本机绝对路径。

### 7.3 确定性与随机性

- 可直接确定的规则必须使用固定输入和精确断言。
- 随机系统必须允许注入或固定随机种子。
- 统计测试必须记录种子、样本量、指标和允许区间。
- 统计区间应根据玩法目标与误差设计，不得为了消除偶发失败而无限放宽。
- 平衡观察与正确性门禁必须分开：资金守恒属于正确性，NPC 平均退契率属于平衡区间。

### 7.4 断言边界

优先断言稳定契约：

- 资金总量、物品总量、时间推进次数。
- 状态转移是否合法、是否最终收敛。
- 玩家可执行的入口与最终反馈是否存在。
- 同一业务事实是否只有一个来源。

避免直接绑定易变实现：

- 私有节点的偶然层级。
- 完整长文案逐字匹配。
- 非必要的像素坐标。
- NPC 必须在某一固定毫秒行动。

当 UI 结构本身就是设计承诺，例如“结算只在底部出现一次”，可以对结构或语义标签进行精确断言。

### 7.5 动画与对话

- 动画测试必须验证真实帧序列、动画名称、播放方向和结束状态，不以简单缩放或漂浮替代角色动画。
- NPC 行动停顿测试应验证事件顺序和允许时间区间，不依赖单个固定延迟。
- 每个行动事件必须拥有唯一 ID；主台词、表情和动作均挂在该 ID 下，防止同一行动重复发言。
- 已退契、离桌、昏迷或失去资格的角色不得继续进入思考、动作或对话队列。

### 7.6 视觉测试

- 截图必须固定 Godot 版本、操作系统、字体、窗口尺寸、缩放和游戏状态。
- 截图脚本首先负责生成评审证据；只有基线稳定后才可以升级为阻断门禁。
- 像素差异必须设置合理容差，并保留差异图，不能只输出“截图不同”。
- 美术品质仍需人工评审；视觉自动化重点检查缺图、错图、裁切、遮挡和布局漂移。

## 8. GitHub Actions 配置细则

具体 Workflow 必须遵守以下约束：

1. 文件只能放在仓库根目录 `.github/workflows/`。
2. 第一阶段触发条件为：推送到 `main`、针对 `main` 的 Pull Request、手动触发。
3. 默认权限必须为 `contents: read`；没有明确用途不得授予写权限。
4. Godot 必须固定到项目声明的精确稳定版本，升级必须单独提交并执行全量测试。
5. 第三方 Action 必须固定完整 commit SHA；优先使用 GitHub 官方 Action 和 Godot 官方发行文件。
6. 不得把密钥写入仓库、日志、测试代码或构建产物。
7. 来自外部分支的测试不得自动获得发布密钥。
8. 每个 Job 必须设置合理超时；每个Godot子进程也必须有独立超时，超时后立即回收并在摘要中写出卡住的测试；第一阶段完整 CI 目标不超过10分钟。
9. 同一分支有新提交时，可以取消仍在运行的旧测试，但不得取消已经开始的正式发布。
10. 缓存只能用于可再生的导入结果或下载内容；不得缓存用户存档、测试结果或秘密。
11. 测试失败必须保留可读日志；视觉任务失败时应上传截图与差异证据。
12. 任何测试命令返回非零退出码时 Job 必须失败，不得使用无说明的 `continue-on-error`。
13. Godot测试即使返回0，只要输出包含`SCRIPT ERROR`、原生崩溃或未处理的`ERROR`也必须失败；单独打印PASS标记不能覆盖此前错误。
14. 主门禁不得依赖开发者自建电脑在线，优先使用 GitHub 托管 Runner。
15. 未提供 `export_presets.cfg` 和导出模板前，不得伪装成“已完成自动构建”。

Workflow 只负责编排，不应在 YAML 中复制大段业务测试逻辑。复杂命令应沉淀为仓库内可在本地和 CI 共同运行的脚本。

## 9. 当前测试基线与归属

第一版 CI 配置编写时，以下测试作为阻断基线：

| 测试 | 层级 | 当前职责 |

当 `smoke_test.gd` 满足任一条件时，相关部分必须拆成独立测试：

- 整体运行超过 3 分钟。

- 新成员无法从测试名判断故障所属模块。

拆分后仍必须保留一个短小的 L3 跨模块闭环，不能把所有测试拆散后失去对整体流程的验证。

## 10. 每次上传前的测试影响审查

每次准备提交并推送游戏修改时，开发者或执行修改的代理必须逐项检查：

```text
[ ] 1. 本次修改涉及哪些设计文档、模块、共享状态和玩家可见行为？
[ ] 2. 是否改变了既有规则、数值、概率、时间、资金、物品或存档口径？
[ ] 3. 是否修复了一个应该永久防止复发的问题？若是，回归测试在哪里？
[ ] 4. 是否新增了状态分支、失败路径、边界值或跨模块接口？
[ ] 5. 是否改变随机算法？种子、样本量和统计区间是否仍合理？
[ ] 6. 是否改变 UI 布局、交互入口、动画、对话或素材？需要哪类结构/视觉测试？
[ ] 7. 是否新增、移动或删除资源？所有路径和图集契约是否有验证？
[ ] 8. 是否改变存档字段或默认值？是否需要迁移测试？
[ ] 9. 现有测试是否仍表达设计意图，而不是仅仅复刻当前实现？
[ ] 10. CI 的 Godot 版本、命令、权限、缓存、超时或触发范围是否需要调整？
[ ] 11. 已运行哪些本地测试，结果是什么？未运行的原因是什么？
[ ] 12. 若判断无需修改测试，理由是否明确且可复核？
```

审查结论至少分为三种：

- **需要新增测试**：与实现一起完成后才可上传。
- **需要更新测试或 CI**：同一提交或同一组连续提交完成，不允许长期失配。
- **无需修改测试**：记录理由，仍运行与修改范围匹配的既有基线。

不得因为“改动很小”“本机玩起来正常”或“以后再补”而跳过影响审查。

## 11. 失败、豁免与不稳定测试

### 11.1 CI 失败

CI 失败后只能有三种处理：

1. 修复实现。
2. 证明设计已改变，并同步修改设计和测试。
3. 证明属于环境或测试缺陷，并修复测试基础设施。

禁止直接删除断言、扩大随机区间或跳过整个 Job 来获得绿色状态。

### 11.2 临时豁免

只有满足以下条件才允许临时非阻断：

- 失败与当前功能正确性无关。
- 已创建明确的修复记录。
- 写明负责人、原因和恢复阻断的条件。
- 仍然执行并保存失败证据。

视觉截图在基线尚未稳定时可以非阻断，但规则守恒、存档安全、资金和物品正确性不得豁免。

### 11.3 不稳定测试

同一测试在相同提交、相同环境中出现不一致结果，即视为不稳定测试。处理顺序为：

1. 固定随机源、时钟、异步事件或资源加载条件。
2. 缩小测试职责并补充诊断输出。
3. 必要时暂时移出阻断门禁，但继续单独运行和记录。
4. 不得通过自动重跑后取成功结果来掩盖不稳定性。

## 12. 分阶段落地顺序

### 阶段 A：第一版持续测试

- 固定 Godot 4.7 stable。
- 加入 L0 导入检查。
- 运行第 9 节列出的七个阻断测试。
- 支持 `push`、Pull Request 和手动触发。
- 使用只读权限并保留完整日志。

### 阶段 B：模块化与资源验证

- 将持续膨胀的烟雾测试拆为模块测试。
- 建立存档往返和迁移测试。
- 建立素材路径、图集尺寸和动画帧检查。
- 稳定两个截图脚本并生成可审阅产物。

### 阶段 C：自动构建

- 提交并审查 `export_presets.cfg`。
- 安装与 Godot 版本完全一致的导出模板。
- 生成 Windows 测试包并执行启动检查。

### 阶段 D：正式发布

- 仅由版本标签或人工批准触发。
- 全量测试通过后生成带版本号的压缩包。
- 记录 commit、Godot 版本、构建时间和校验和。
- 发布流程与普通 PR 测试分离，避免误发布。

## 13. 完成定义

一项游戏修改只有同时满足以下条件，才算完成：

1. 行为与对应设计文档一致。
2. 已完成测试影响审查。
3. 新规则、新分支和已修缺陷拥有相称的测试。
4. 本地相关测试通过。
5. 推送后的阻断 CI 通过。
6. 需要人工判断的美术、手感或平衡问题已留下评审证据，而不是被错误包装成自动化结论。
7. 测试和 CI 没有依赖个人电脑、真实存档或未声明的秘密条件。

“CI 通过”不是质量工作的终点，但它必须意味着：这个版本没有破坏项目已经正式承诺并能够客观验证的内容。

## 14. 宪法维护

出现以下情况时必须重新审阅本文档：

- 新增大型玩法、在线功能或外部服务。
- 引入正式存档版本、模组、DLC 或多平台发布。
- Godot 主版本升级。
- CI 总时长持续超过 10 分钟。
- 不稳定测试或误报开始影响正常开发。
- 同类线上/试玩缺陷重复出现，说明现有测试契约不足。

修改本宪法时必须在版本记录中说明：修改原因、影响的测试层级、是否需要同步调整 Workflow。
